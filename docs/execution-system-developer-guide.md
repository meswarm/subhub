# 执行系统接入 Pulse — 开发者须知

> **读者对象：** 你有一个工具或系统，想让它可以通过 Pulse 被用户调用。本文告诉你 **需要提供什么**。
>
> ⚠️ **你不需要编写 manifest.yaml、SKILL.md 或任何 Pulse 配置文件。** 这些由 Pulse 管理员完成。
>
> ⚠️ **你不需要改变系统的内部架构。** Pulse 只需要一个 CLI 入口来调用你的系统，你的系统内部怎么工作保持不变。

---

## 核心原则

**Pulse 不替代你的系统，只是调用它。**

- 如果你的系统 **没有 AI**（纯工具），Pulse 的中间件会用自己的 AI 来理解用户意图，然后调用你的 CLI
- 如果你的系统 **已有 AI**（自带大模型），Pulse 中间件只做转发，把用户原文直接传给你的 CLI，由你的系统自己处理

无论哪种情况，你都只需要提供一个 **CLI 入口** — 让 Pulse 能通过命令行调用到你的系统即可。

---

## 你需要提供的内容

### ① CLI 入口

一个可以通过命令行调用的可执行文件。**不需要新写一个 CLI，在你现有系统上加一个命令行入口即可。**

根据你的系统类型，CLI 的设计方式不同：

#### 情况 A：系统已有 AI（自带大模型处理）

你的系统已经能理解自然语言了。CLI 只需要接受一段文本，交给你系统内部的 AI 处理，返回结果：

```bash
# 你的 CLI 接受用户原文，内部调用你自己的大模型处理
./subhub "我有哪些订阅快到期了？"
# 输出: 以下订阅将在 7 天内到期:
#   - Netflix (2026-04-15)
#   - Spotify (2026-04-17)

./subhub "帮我记录一个新订阅 Netflix 每月 15.99 美元"
# 输出: ✅ 已创建订阅: Netflix $15.99/月
```

**关键点：**
- CLI 内部仍然使用你自己的大模型来理解意图和执行操作
- Pulse 中间件不会替代你的 AI，只是把用户原文透传给你
- 你的系统的智能、记忆、上下文能力完全保留
- 不需要把功能拆成 `create`、`list`、`delete` 等子命令（除非你本来就有）

#### 情况 B：系统没有 AI（纯工具 / CLI 工具）

你的系统是一个纯命令行工具，不理解自然语言。这时需要设计结构化的子命令，Pulse 的中间件 AI 会学习如何调用它们：

```bash
# 结构化子命令 + 参数
./todo-cli create --content '明天要交论文' --deadline '2026-04-08'
./todo-cli list
./todo-cli complete --id 'abc12345'
./todo-cli delete --id 'abc12345'
```

**子命令设计要求（仅适用于没有 AI 的系统）：**
- 每个功能用子命令区分
- 使用 `--name value` 参数格式
- 禁止交互式提示（不能等 stdin 输入）

### ② 命令说明文件：`<system_id>-manual.md`

提供一份以你的 `system_id` 为前缀的 Markdown 说明文件，命名格式为 **`<system_id>-manual.md`**。

> `system_id` 由 Pulse 管理员分配。如果尚未确认，先与管理员沟通获取。
> 示例：系统 `service.subhub` 对应文件名为 `service.subhub-manual.md`。
> 仅把pulse需要的知道的说明写在文档, 不需要知道的不用写

Pulse 管理员会根据这份说明来编写 Pulse 侧的配置文件。

**自带 AI 的系统** 需包含：
```
1. CLI 调用方式（如 ./subhub "<用户文本>"）
2. 工作目录（CLI 应在哪个目录下执行）
3. 系统能做什么（功能范围描述）
4. 输出格式说明
5. 特殊注意事项（如需要的环境变量等）
```

**没有 AI 的系统** 需更详细：
```
1. CLI 调用方式（如 ./todo-cli <subcommand>）
2. 工作目录
3. 所有子命令列表
4. 每个子命令的参数（哪些必填、哪些可选、格式要求）
5. 每个子命令的成功输出和错误输出示例
6. 操作上的依赖关系（如"删除前必须先查询获取 ID"）
7. 特殊注意事项
```

### ③ 主动事件调用（仅在需要主动通知用户时）

如果你的系统需要 **主动** 给用户发通知（如到期提醒、异常告警），在代码中调用 Pulse Event Gateway 的 HTTP 接口。具体见本文末尾"主动事件"一节。

> 如果你的系统只做被动任务（只等用户调用），则只需要提供 ① 和 ②。

---

## 你不需要做的事

以下工作全部由 Pulse 管理员完成，**你不需要做**：

- ❌ 编写 SKILL.md（AI 代理操作手册）
- ❌ 编写 manifest.yaml（中间件配置）
- ❌ 注册 system_id 到 registry
- ❌ 理解 NATS / Matrix / Pulse 内部协议
- ❌ 替换或放弃你系统内部的大模型
- ❌ 把你的系统拆成"纯业务 CLI"去掉 AI 能力

---

## CLI 通用输出规范

无论哪种类型的系统，CLI 输出需要遵守：

- 成功输出 → 打印人类可读的结果文本到 stdout，退出码 0
- 错误输出 → 以 `Error:` 开头到 stderr，退出码非 0
- 编码 → UTF-8
- 不要输出 → 调试日志、进度条、纯 JSON（用户直接看不懂）

## 主动事件（仅在需要主动通知用户时）

如果你的系统需要 **主动** 向用户发消息（不是等用户问，而是系统自己触发，如到期提醒），在代码中调用以下 HTTP 接口。

> ⚠️ 使用前请先与 Pulse 管理员确认你的 `system_id`。这个 ID 由管理员分配。

### 发通知

```bash
curl -X POST http://127.0.0.1:8091/events/notify \
  -H 'Content-Type: application/json' \
  -d '{
    "system_id": "<你的system_id>",
    "message": "你的 Netflix 订阅明天到期"
  }'
```

### 发告警

```bash
curl -X POST http://127.0.0.1:8091/events/alert \
  -H 'Content-Type: application/json' \
  -d '{
    "system_id": "<你的system_id>",
    "message": "磁盘空间不足 3%"
  }'
```

### 向用户提问（等待回复）

```bash
curl -X POST http://127.0.0.1:8091/events/ask-user \
  -H 'Content-Type: application/json' \
  -d '{
    "system_id": "<你的system_id>",
    "question": "订阅即将到期，是否续费？",
    "timeout": 300
  }'
```

**响应格式：**

| 场景 | 响应 |
|------|------|
| 发送成功 | `{"status": "sent", "subject": "...", "task_id": "..."}` |
| 用户回复 | `{"status": "asked", "reply": "续费", "thread_id": "..."}` |
| 用户超时 | `{"status": "timeout", "thread_id": "..."}` |
| system_id 无效 | HTTP 403，`{"error": "unknown system"}` |
| 事件类型不在白名单 | HTTP 403，`{"error": "event type not allowed"}` |

## 常见问题

**Q: 我的系统已经有 AI 了，Pulse 会替代它吗？**
不会。Pulse 中间件只是把用户说的话转发给你的 CLI，你系统内部的 AI 正常工作。Pulse 不会在中间加一层自己的 AI 来"重新理解"你的输入。

**Q: 我需要新写一个 CLI 吗？**
不一定。如果你的系统已经有命令行入口（或者可以很容易加一个），直接用它就行。核心要求是：能通过命令行传入文本、返回文本。

**Q: 我的工具需要读写文件，有安全限制吗？**
Pulse 有命令白名单机制，只有管理员配置的命令前缀才会被允许执行。你不需要操心这个。

**Q: 执行时间很长怎么办？**
告知 Pulse 管理员你的 CLI 预计执行时间，管理员会调整超时配置。

**Q: system_id 是什么？从哪里来？**
这是你的系统在 Pulse 中的唯一标识，由 Pulse 管理员分配。你 **只在调用主动事件接口时** 需要用到它。如果你的系统只做被动任务，你甚至不需要知道这个 ID。
